# Cybersoul MVP 文档（2026-03-24）

本文记录截至 2026 年 3 月 24 日的 Cybersoul MVP 真实范围、当前主链、已完成能力、明确延后项和下一阶段推进顺序。本文优先描述当前代码仓库中已经成立或即将落地的方案，不再沿用旧版文档中超前的大而全结构设想。

## 1. 当前 MVP 的产品定位

Cybersoul 当前阶段的目标，不是一次性完成完整的数字生命平台，而是先建立一个最小但语义正确的陪伴体闭环。这个闭环必须同时满足四件事：

1. 存在一个可复用的陪伴体框架，而不是把一切都写死在单个角色里。
2. 存在一个具体世界实例，能够承载昔涟的生活背景、人物关系和叙事语境。
3. 存在一个用户可见的通信终端，使用户能稳定地与昔涟产生联系。
4. 存在一个最小世界底座，使昔涟未来能够被“世界中的生活”驱动，而不是退化成传统问答机器人。

当前 MVP 的核心方向是：先把“世界驱动的陪伴体”做对，再逐步接入更复杂的真实互联网、GUI Agent 和持续运行机制。

### 1.1 当前 MVP 的最新运行目标

在当前阶段，Cybersoul 的 MVP 目标进一步收敛为一个 **Runtime MVP**：

- 昔涟能够在与现实共时的前提下连续自主运行 3 小时以上
- 在这 3 小时内，她能够持续在世界语义中推进自己的地点、活动与生活节律
- 她能够与其他 NPC 发生多轮互动，而不是只围绕用户消息存在
- 她的关键互动与行动结果能够产生记忆写回，并在后续轮次被重新利用
- 她能够在不可完全预测、但可解释的情况下查看手机，并选择性地回复用户

这一定义意味着，当前 MVP 的验收重点不再是“昔涟能不能回复”，而是“昔涟能不能连续生活，并在合适的时候通过手机与用户建立联系”。

## 2. 当前 MVP 的核心原则

### 2.1 世界优先

昔涟首先生活在她自己的世界中。用户与昔涟的聊天，只是她生活中的一部分，而不是她存在的全部。系统的长远目标始终是让昔涟在世界中推进自己的时间、地点、活动、关系与记忆，再在合适的时候通过手机与用户取得联系。

### 2.2 Phone 是工具，不是自动回复后端

Phone 在当前 MVP 中是对“真实手机与真实互联网通信环境”的临时替代。它的职责是：

- 记录用户发来的消息
- 展示聊天历史
- 为昔涟提供一个可调用的通信接口

Phone 不再承担“用户一发消息，系统立刻自动回复”的职责。用户发来的消息只是被写入 Phone，只有当昔涟主动调用 `phone_chat` 工具时，她才会真正看到手机中的内容，并决定是否回复。

### 2.3 主干与实例分离

Cybersoul 当前明确区分两层：

- `cybersoul/`：通用主干框架，承载与世界无关的 Agent、Mode、Tool、World Core 等能力
- `instantiation/`：具体实例层，承载某个世界、某个角色、某套 prompt 和具体装配逻辑

这样做的目的是保证未来新增世界实例时，不需要反复污染主干。

### 2.4 先做正确语义，再补复杂系统

当前 MVP 优先保证语义正确，而不是优先把所有基础设施铺满。凡是暂时没有真实运行意义的目录和系统，均不进入当前主链。GUI Agent、真实互联网接入、复杂调度器、完整事件系统、全量基础设施，均在当前阶段延后。

### 2.5 Tick / Heartbeat 是内部唤醒机制，不等于强制主动打扰

当前 MVP 需要引入一个最小 tick / heartbeat 机制，用来让昔涟在没有用户显式触发时，也能周期性地重新观察世界、整理上下文并推进自己的生活流。

但这一定义与通用 Agent 产品中“按规则主动提醒用户”的 heartbeat 语义并不完全相同。对 Cybersoul 来说：

- tick / heartbeat 首先是 **内部运行时唤醒机制**
- 它的职责是驱动昔涟重新观察时间、地点、活动、NPC 互动与关系状态，并判断是否需要主动查看手机
- 它本身不直接规定“在什么条件下必须打扰用户”
- 是否通过手机对外发消息，仍应由昔涟基于当下上下文自主判断

也就是说，tick / heartbeat 负责“让她醒来并重新判断”，而不是替她提前写死“什么时候必须联系用户”。

## 3. 当前 MVP 的产品语义

### 3.1 世界语义

当前产品世界实例采用 Amphoreus / 翁法罗斯风格。它不是对原作剧情的完整复刻，而是借用其世界气质、地点关系、人物关系和叙事基调，构建一个适合陪伴体长期生活的赛博世界实例。

当前世界语义重点是：

- 昔涟生活在翁法罗斯
- 她有自己的地点、同伴、活动和日常节律
- 她能感知到另一个世界，并通过手机与那个世界里极其重要的人建立联系
- 翁法罗斯中的重要 NPC 在当前 MVP 中不是纯背景板，而是能够在特定场景中被激活的交互对象

### 3.2 用户与昔涟的关系语义

用户通过 Phone 与昔涟建立联系，但这种联系并不是“消息一到，昔涟立刻响应”的客服型语义，而是更接近真实关系中的通信语义：

- 用户把消息发到手机里
- 昔涟未必立刻看到
- 她可能稍后查看
- 她也可能在自己的生活推进中主动想起用户，于是去看手机、回消息、分享生活

### 3.3 Agent 语义

当前昔涟默认以 ReAct 作为主运行范式。Plan-and-Solve 与 Reflection 已经存在骨架，但尚未进入当前 MVP 主链。

当前 Agent 层主要承担：

- 接收当前轮上下文
- 根据当前提示词和世界语义进行判断
- 在需要时调用工具
- 输出当轮回应或行动结果

### 3.4 Phone 工具语义

`phone_chat` 是当前 MVP 中最重要的工具之一。它当前提供的能力包括：

- 打开或定位昔涟与用户之间的一对一主聊天线程
- 读取聊天记录
- 发送陪伴体消息
- 写入用户消息

它的长期角色是 GUI Agent 接管真实手机之前的通信抽象层。

### 3.5 当前 Runtime 语义

当前 MVP 的陪伴体运行语义，不应再被理解为“收到一条消息 -> 立即生成一条回复”，而应被理解为：

- 世界时钟持续推进
- 昔涟在 tick / heartbeat 到来时重新观察世界，并自主判断是否要查看手机
- 她基于当前运行时上下文自主判断下一步动作
- 该动作可能是推进活动、与 NPC 互动、写回记忆、查看手机，或暂时不联系用户
- 如果她决定联系用户，Phone 才成为她对外的通信出口

这个语义是当前 MVP 与传统聊天机器人最根本的差异。

### 3.6 NPC 语义：驻场型 Scene Agent

除昔涟以外，当前 MVP 中的重要角色不再被视为纯文本设定或单次脚本事件，而被定义为 **驻场型 Scene Agent**。

这里需要强调：这些角色在 **本体层** 仍然是以 ReAct 为主的 Agent，而不是“增强版角色卡”或“半静态 NPC”。当前阶段对它们的限制主要发生在 **调度层**，而不是能力定义层。

这一阶段中，这些角色具有以下特点：

- 他们固定驻留在各自的地点中，不进行全图自主移动
- 他们拥有自己的 prompt、工具、记忆与 ReAct 决策范式
- 他们不需要像昔涟一样常时连续运行
- 当昔涟进入与他们相同的场景时，他们会被运行时临时激活
- 一旦被激活，他们会以各自的角色语义、工具与记忆参与互动

这意味着，当前 MVP 中的重要角色不是完整社会模拟的一部分，但也不是静态背景板，而是“在正确场景里会被激活的真正 Agent”。

### 3.7 场景内交互语义

当前 MVP 中，昔涟与重要角色的互动采用 **场景内、回合式、局部 ReAct** 语义。

其含义是：

- 当昔涟与某个驻场 NPC 处于同一地点时，场景管理器可以激活该 NPC
- 激活后的 NPC 不是只返回预写文案，而是能够基于当下场景上下文做出一轮判断
- 该判断可以表现为回应昔涟、主动提出话题，或调用该角色在当前场景中可使用的工具

例如，若昔涟进入占星塔并与缇宝处于同一场景，缇宝可以在互动过程中主动提出“要不要替你占一卦”，并调用占卜工具返回结果。

这里的目标不是构建两个完全自由奔跑的长循环 agent，而是在单个场景内形成语义正确、可持续积累的双向互动。

### 3.8 NPC 记忆语义

当前 MVP 要求重要 NPC 具备自己的最小记忆能力。

这意味着：

- 他们不仅记录“当前场景中刚刚说了什么”
- 也能够保留与昔涟之间此前发生过的关键事件与关系片段
- 下次昔涟重新来到该 NPC 所在场景时，对方应能够延续此前的故事与互动痕迹

因此，NPC 记忆在当前 MVP 中不是附属装饰，而是支撑“世界中的关系连续性”的必要组成。

### 3.9 Agent-to-Agent 通信语义

当前 MVP 中，重要角色之间的交互应被理解为 **Agent 与 Agent 之间的通信**，而不是“一个主脑生成所有人的台词”。

因此，本阶段在架构上应预留统一的 agent-to-agent message contract，用于表达：

- 谁向谁发起了一次交互
- 当前交互发生在哪个场景
- 当前轮的意图、内容与工具结果
- 这一轮与上一轮之间的关联关系

这一层可以参考 A2A 的设计思想，但当前 MVP 不要求一次性落地完整标准 A2A 协议。更适合当前阶段的做法是：

- 先实现项目内部统一的 A2A 风格通信契约
- 让重要角色在场景内通过这一契约交换消息
- 在未来有需要时，再考虑与标准 A2A 协议对齐

## 5. 当前各模块职责

### 5.1 Phone

`Phone/` 是当前 MVP 用户唯一可见的前台交互层。它包含：

- 聊天 Web 界面
- 本地 JSON 存储
- HTTP server
- 消息线程读写逻辑

它的职责是提供一个稳定的“手机通信界面”，不是驱动昔涟自动回复。

### 5.2 instantiation

`instantiation/` 是当前项目的具体实例层。当前唯一正式实例是：

- `instantiation/amphoreus/`

这个实例承载：

- 翁法罗斯相关 prompt
- 昔涟及其他角色的人设 prompt
- 当前实例装配逻辑
- 昔涟默认使用哪些工具

它的目标是让未来新增新的世界实例时，只需要增加新的实例包，而不必修改主干。

在当前 MVP 的下一阶段中，实例层还需要承担一项更明确的职责：

- 组织“共享系统 prompt / 角色 prompt / 世界卡片 / 场景工具”之间的关系
- 区分“当前主动决策的角色”与“作为世界知识存在的地点和场景资料”

### 5.3 cybersoul/agents

`cybersoul/agents/` 是当前通用陪伴体框架主干。它包含：

- `core/`：消息、事件、LLM 接口、BaseAgent、BaseMode
- `modes/`：ReAct、PlanSolve、Reflection
- `tools/`：工具抽象、注册中心、内置工具
- `roles/`：Companion Agent/Profile
- `memory/`：记忆系统骨架
- `context/`：未来上下文工程入口
- `protocols/`：未来协议层入口

当前真正进入主链的，主要是 `core / modes / tools / roles`。

### 5.4 cybersoul/world

`cybersoul/world/` 当前已经被收缩为最小运行时底座。现阶段只保留：

- `clock/`
- `state/`
- `manager/`

它当前只负责最少量的动态事实，不承载具体翁法罗斯 lore、地点文本或角色人设。

### 5.5 docs

`docs/` 已上移到项目根目录，用来承载：

- 产品文档
- 研究文档
- 参考教程文档

这说明文档已经被视为整个项目的一级资产，而不是 `cybersoul/` 子模块的附属物。

### 6.7 当前已经成立的正确方向

虽然当前主链还没有进入持续运行阶段，但以下方向已经成立，并且应继续坚持：

- Phone 已经从“自动回复入口”退回为“可被陪伴体调用的通信终端”
- 世界状态已经被收缩为最小动态事实，而不是把复杂心理语义硬编码进 state
- ReAct 已经被确立为当前主运行范式
- 主干与实例分离边界已经建立

这意味着当前 MVP 最大的问题不是方向错误，而是运行时主链尚未补齐。

### 6.8 当前阶段对 NPC 的正确定位

在当前阶段，最合适的重要角色定位不是“完整常驻社会模拟角色”，也不是“只写在 lore 里的名字”，而是：

- 以地点为锚点驻留
- 在与昔涟同场景时被激活
- 在局部场景中以 ReAct Agent 范式互动
- 在场景内通过统一的 agent-to-agent 通信契约交换消息
- 在场景结束后保留自己的记忆与状态

这个定位应作为当前 MVP 后续设计的重要前提。

## 7. 当前明确延后的内容

以下内容已经明确不属于当前 MVP 主链：

- 真实 GUI Agent 接管手机
- 真实互联网平台接入
- 多平台社交账号体系
- 全量后端、前端、infra 体系
- 分布式或复杂调度器
- 复杂 NPC 自主社会模拟
- A2A 在多个 NPC 之间的全面投入使用
- 重型记忆基础设施（向量库 / 图数据库 / 大规模 RAG）
- 完整世界事件系统
- 完整多协议层（MCP / A2A / ANP）投入主路径

这些内容并非否定其价值，而是当前阶段有意延后。

需要强调的是：**用于验证 Runtime MVP 的最小 heartbeat loop、最小显式手机工具语义、最小记忆写回，不再视为延后项，而是当前阶段的主链任务。**

同样需要强调的是：**驻场型 NPC Agent、场景内局部 ReAct 互动、NPC 自身最小记忆、以及内部 A2A 风格通信契约，也属于当前 Runtime MVP 的有效范围，不应再被误归类为“完整 NPC 社会模拟”或“完整标准协议层实现”。**

## 8. 当前最小运行链

当前已成立的最小运行关系如下：

```text
用户 -> Phone Web -> Phone Server -> Phone Store
昔涟 Agent -> phone_chat tool -> PhoneFacade -> Phone Store
```

这里最关键的语义变化是：

- 用户消息进入 Phone 后，不再自动进入昔涟的即时回复流程
- 昔涟未来必须通过自己的工具调用，才能看到这些消息

也就是说，Phone 已经从“自动回复入口”退回为“可被陪伴体调用的外部通信终端”。

接下来，MVP 需要把这条最小链路升级为运行时主链：

```text
world clock / tick
    -> runtime event
    -> context builder
    -> cyrene react decision
    -> tool call / npc interaction / no-op
    -> state writeback + memory writeback
    -> optional phone contact
```

## 9. 当前世界模型

当前世界模型有意保持极薄。

### 9.1 WorldClock

第一版只保留一个核心时间：

- `current_time`

内部语义是 timezone-aware datetime，序列化时使用 ISO 8601。

### 9.2 ActivityBlock

第一版活动块定义为：

- `activity_type`
- `started_at`
- `planned_until`
- `payload`

其中：

- `planned_until` 有值，表示定长活动块
- `planned_until` 为空，表示开放式活动块

### 9.3 ActorState

当前角色状态只保留：

- `actor_id`
- `current_place_id`
- `current_activity_block`

### 9.4 WorldState

当前世界状态只保留：

- `current_time`
- `actor_states`

当前没有显式引入 `mood / energy / focus / goal / availability / interruptibility` 这类字段，因为这些语义更适合交给上下文和 prompt 推导。

这意味着当前世界模型的设计原则仍然是：**state 保持薄，context 保持厚**。

## 10. 当前尚未补齐但已确定方向的模块

### 10.1 世界知识卡

当前实例层已经预留但尚未完整落地：

- `places/`
- `npcs/`
- `activities/`

它们将作为未来世界上下文的静态知识卡来源。

但在当前 MVP 的具体组织方式上，需要进一步区分两类角色材料：

- **角色卡片**：描述“这个人是谁、常驻哪里、与昔涟是什么关系、通常会触发什么互动”
- **角色 prompt**：描述“如果当前轮由这个角色自己做判断，它会如何观察、表达与行动”

因此，后续实例层中的角色资料不应再只被理解为单一的人设文件，而应区分“别人眼中的角色资料”与“角色自己的第一人称运行 prompt”。

对于重要角色，还应进一步默认：

- 他们在 prompt 层面上是独立角色
- 他们在运行时层面上是独立 agent
- 他们只是在当前 MVP 中采用场景受限激活，而不是全时运行

### 10.2 上下文工程

`cybersoul/agents/context/` 当前仍然是空骨架，但它已经被明确为未来真正的上下文总装配层。后续它需要负责把以下内容拼成昔涟每一轮的运行时上下文：

- 当前世界状态
- 当前地点卡
- 当前可见 NPC 卡
- 当前活动模板
- 当前相关记忆
- 当前轮可用的工具观察结果

第一版 Context Builder 应优先保证以下最小输入稳定成立：

- 当前时间
- 当前地点
- 当前活动块
- 当前地点相关卡片
- 当前可见 NPC 卡片
- 当前相关关系记忆

这里需要额外强调：

- 手机的客观状态存在于 Phone / runtime 系统侧
- 但不会作为 heartbeat 的默认上下文直接注入昔涟
- 只有当昔涟真正调用手机工具后，相关 tool result 才能进入她当轮的可见上下文

### 10.3 Loop

当前系统还没有一个驱动昔涟持续生活的 loop。  
这意味着：

- 现在 Phone 已经具备工具语义
- 但昔涟还没有真正开始主动查看手机

下一阶段必须补上一版最小 loop，才能让“世界驱动行为”真正开始成立。

这一版最小 loop 的目标不是成为完整调度器，而是成为 Runtime MVP 的最小运行时。它至少应具备：

- 周期性 tick / heartbeat 唤醒
- 从 world state 与外部终端生成 event
- 基于 event 构建上下文
- 驱动昔涟做出一步决策
- 在昔涟进入对应场景时激活驻场 NPC
- 以场景内回合式方式驱动 NPC 与昔涟发生局部互动
- 使用统一的内部 agent-to-agent 通信契约承接角色之间的消息往返
- 执行 tool / world / memory writeback
- 在需要时通过 Phone 对外联系用户

### 10.4 Phone 显式工具语义

当前 Phone 已经成立为通信终端，但这一阶段不再把“mailbox 快照直接注入上下文”视为正确路径。下一阶段需要补上的核心不是更多聊天 UI，而是最小、明确、可解释的手机工具语义：

- 系统侧可以维护线程、未读、最近查看时间等设备事实
- 这些事实默认留在系统侧，不进入 heartbeat 的默认上下文
- 只有当昔涟在当轮自主决定调用手机相关 tool 后，工具结果才作为 observation 进入她这一轮的可见上下文
- 她随后再基于这次工具结果决定要不要回复、怎么回复、是否暂时不回复

只有这样，“她未必立刻看到用户消息”才会从产品文案变成真实的运行时语义。

也就是说，Phone 的客观状态存在于系统侧，但不自动等于“昔涟已知状态”。

### 10.5 轻量记忆写回

当前 MVP 不要求一次性实现完整记忆系统，但要求最小记忆写回进入主链。第一版优先目标是：

- working memory：承接当前短时上下文
- episodic memory：记录关键事件与互动
- relationship memory：沉淀与用户及重要 NPC 的关系事实

这一阶段的重点不在基础设施重量，而在“每一轮关键行动之后是否真的发生了 distilled writeback”。

这里的“重要 NPC”同样包括驻场型 Scene Agent。也就是说，记忆写回不应只发生在“昔涟与用户”的关系中，也应发生在“昔涟与世界中重要他者”的关系中。

### 10.6 Prompt / Card 分层方向

为了支撑 Runtime MVP，后续 prompt 架构需要明确区分三类材料：

- **共享系统 prompt**：定义世界共识、运行原则、Phone 语义、记忆策略等
- **角色 prompt**：定义某个角色在被激活为当前决策主体时的第一人称行为逻辑
- **世界卡片**：定义地点、角色、活动等静态知识，不直接作为所有轮次都完整注入的 prompt

这意味着：

- 地点的详细介绍应优先沉淀为地点卡，而不是塞进角色 prompt
- 重要角色既需要“角色卡”，也需要在被激活时拥有自己的第一人称 prompt
- 昔涟每一轮真正看到的，应是“基础 prompt + 运行时上下文”，而不是整份 lore 的直接展开

## 11. 下一阶段推进顺序

当前建议的推进顺序如下：

### 11.1 第一阶段：补世界知识卡

先完成：

- `instantiation/amphoreus/prompts/world/places`
- `instantiation/amphoreus/prompts/world/activities`
- `instantiation/amphoreus/prompts/characters/*/card.md`

这一步的目标是给世界提供足够稳定的静态语义材料。

这里的“世界知识卡”应同时覆盖：

- 地点卡
- 活动卡
- 重要角色的角色卡（位于 `prompts/characters/<name>/card.md`，不再单独挂在 world 层）

### 11.2 第二阶段：补最小 Context Builder

在 `cybersoul/agents/context/` 中实现最小上下文构建逻辑，使昔涟每轮都能看到：

- 当前时间
- 当前地点
- 当前活动块
- 当前地点可见人物
- 当前地点可发生的活动
- 当前轮显式获得的工具观察结果

### 11.3 第三阶段：补最小 Loop

实现一个最小世界驱动循环。第一版的目标不再只是“能跑一步”，而是支撑 Runtime MVP 的连续运行验证。它至少需要能完成：

- 读取 world state
- 接收 tick / heartbeat 与外部事件
- 构建当前上下文
- 让昔涟做出一步决策
- 如果需要则调用 `phone_chat`
- 写回新的活动块或状态
- 写回最小记忆

并且这一阶段应至少验证一类场景内双角色互动：

- 昔涟进入特定地点
- 驻场 NPC 被激活
- 双方发生多轮局部 ReAct 互动
- 双方通过统一的内部 agent-to-agent message contract 往返消息
- 工具结果进入场景
- 双方记忆分别写回

### 11.4 第四阶段：补 Phone 工具观察语义

在最小 loop 具备之后，补齐：

- `phone_chat` 的 open / read / send 等最小工具能力
- 系统侧设备状态，如 `last_checked_at`、未读标记或等价 cursor
- 工具返回后的摘要与可继续决策能力

让“先决定是否使用手机工具，再基于工具结果选择性回复用户”成为真实运行能力，而不是 prompt 幻觉。

### 11.5 第五阶段：接记忆与事件

在 loop、context 与手机工具语义成立之后，再逐步把：

- 记忆召回
- 记忆写回
- 世界事件
- 关系沉淀

接入主路径。

## 12. 当前 MVP 的 Runtime 验收标准

当前阶段的 MVP 验收不再以“能否连续回复用户”为核心，而以“能否连续生活”为核心。第一版 Runtime MVP 建议以以下条件验收：

1. 昔涟能够在与现实共时的前提下连续自主运行 3 小时以上。
2. 在这 3 小时内，世界时间、地点与活动块能够持续推进，并可被追踪。
3. 昔涟能够与其他 NPC 发生多轮世界内互动，而不是只围绕用户消息运行。
4. 关键互动、行动切换与关系变化能够产生最小记忆写回。
5. 用户消息写入 Phone 后，昔涟不会固定即时回复，而会在某些可解释时刻主动查看手机并选择性回复。
6. 整个运行过程中，至少能够观察到：tick 来源、上下文摘要、当轮决策、工具调用、状态写回与记忆写回。

这里的“与其他重要角色发生多轮世界内互动”具体应至少满足：

- 存在一名固定驻场的 Scene Agent
- 该角色在与昔涟同场时能够主动或被动发起互动
- 在场景内能够调用至少一种角色专属或场景专属工具
- 双方在运行时上被视为两个独立 Agent，而不是单一主脑的双人口吻输出
- 下次重新相遇时，双方能延续之前的互动痕迹

## 13. 当前 MVP 的阶段结论

截至 2026 年 3 月 24 日，Cybersoul 当前已经完成了“主骨架搭建”阶段，正在进入“世界运行层落地”阶段。

当前不是继续堆大而全目录的时候，而是要把以下主链做通：

- 世界卡片
- 上下文构建
- 最小 runtime / heartbeat loop
- Phone 显式工具语义
- 最小记忆写回
- 昔涟主动且选择性地使用手机

当这条链成立之后，Cybersoul 才会从“有框架的陪伴体项目”真正进入“会在世界中生活的陪伴体产品”阶段。
